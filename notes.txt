1. Introduction to Containers
Definition

A container is a lightweight, standalone, executable package that includes everything needed to run an application—code, runtime, libraries, and dependencies—while sharing the host OS kernel.

Explanation

Unlike virtual machines, containers do not bundle an entire OS. Instead, they isolate processes at the OS level.

Think of it like this:

VM → Full house with its own land
Container → Apartment in a shared building
Why DevOps cares
Eliminates “it works on my machine” problems
Ensures consistency across environments (dev → test → prod)
Enables rapid deployment and scaling
Origin of Containers

Containers didn’t suddenly appear with Docker.

Key milestones:
1979 – chroot
First attempt at isolating filesystem access in Unix.
2000s – FreeBSD Jails
Added process isolation and networking.
2008 – LXC
Combined namespaces + cgroups → real containerization base.
Emergence of Modern Containerization

The breakthrough came with:

Docker (2013)
Simplified container creation, packaging, and sharing
Introduced:
Image layering
Developer-friendly CLI
Registry ecosystem

Then:

Kubernetes
Automated deployment, scaling, and management of containers
Integration into DevOps

Containers became the backbone of:

CI/CD pipelines
Microservices architecture
Cloud-native systems

DevOps shift:

Before → Deploy code
Now → Deploy containers
2. Container Runtime
Definition

A container runtime is the software responsible for running containers—managing their lifecycle (create, start, stop, delete).

Examples
containerd
CRI-O
Docker (internally uses containerd)
Explanation

The runtime:

Pulls images
Creates isolated environments
Executes the application process

Without a runtime, containers are just static definitions.

3. Process Isolation & Namespaces
Definition

Namespaces are a Linux kernel feature that isolates system resources so that processes inside a container cannot see or interfere with others.

Types of namespaces
PID → process IDs
NET → network interfaces
MNT → filesystem mounts
IPC → inter-process communication
UTS → hostname
USER → user IDs
Explanation

Each container gets its own "view" of the system.

Example:

Inside container → process ID = 1
Outside → it may be process 24567

This illusion is process isolation.

4. Control Groups (cgroups)
Definition

Control Groups (cgroups) limit, prioritize, and monitor resource usage (CPU, memory, disk I/O) for a group of processes.

Explanation

If namespaces provide isolation, cgroups provide control.

Example:

Limit container to:
512MB RAM
1 CPU core

Without cgroups:

One container could consume all system resources → system crash
DevOps relevance
Ensures fairness in multi-tenant systems
Enables predictable performance
5. Container Images & Layers
Definition

A container image is a read-only template used to create containers.

Explanation

Images are built in layers:

Base OS layer (e.g., Ubuntu)
Dependencies layer
Application code layer

Each layer:

Is immutable
Is cached
Can be reused
Example flow:
Base Image → Add Python → Add Libraries → Add App
Why layering matters:
Faster builds
Efficient storage
Reusability across projects
6. Image Registries & Distribution
Definition

An image registry is a centralized storage system for container images.

Examples
Docker Hub
Amazon Elastic Container Registry
Google Container Registry
Explanation

Workflow:

Developer builds image
Pushes to registry
Runtime pulls image
Container runs

This is distribution of software at scale.

Final Mental Model (Important)**

You should internalize this stack:

Application Code
        ↓
Container Image (layers)
        ↓
Container Runtime
        ↓
Linux Kernel Features:
   - Namespaces (isolation)
   - cgroups (resource control)
        ↓
Infrastructure (VM / Cloud / Bare Metal)
